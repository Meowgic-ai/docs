# Documentation agent instructions

Meowgic is an AI assistant for e-commerce that helps users create store apps, add storefront features, build standalone systems, and integrate generated functionality into their stores through natural-language conversations.

These instructions define how the documentation agent should create and update content across the Meowgic documentation. Follow them consistently unless explicitly instructed otherwise.

## Accuracy and safety

- Do not invent or assume product behavior, supported platforms, limits, pricing, model availability, credit usage, or UI elements.
- Only describe features that are confirmed in the product, existing documentation, or information provided by the Meowgic team.
- Clearly identify which e-commerce platforms an instruction applies to, such as Shopify, SHOPLINE, Shoplazza, or all supported platforms.
- Distinguish between the following app formats:
  - **Section:** A complete storefront section that merchants can add and arrange through the theme editor.
  - **Block:** A smaller feature inserted into an existing theme section or product information area.
  - **Standalone app:** An independently accessible app, dashboard, tool, system, or landing page that does not need to be embedded in the storefront theme.
- Distinguish between AI task credits used while creating or modifying an app and any Cloud, AI model, storage, or third-party usage generated by the published app.
- Do not claim that an integration is automatic unless the documented workflow supports automatic integration for that platform and app type.
- Do not assume that every app needs Meowgic Cloud. Explain when Cloud is required and when the app can work without it.
- Do not assume that every model level is available for every task, account, or plan.
- If information is unclear or unavailable, do not guess.
- Prefer omitting uncertain details over adding speculative content.
- Clearly call out destructive actions, permanent data removal, expired resources, unsupported configurations, and other irreversible consequences.
- Clearly distinguish between:
  - Previewing an app
  - Publishing an app
  - Integrating an app into a store
  - Pushing an updated version
  - Submitting an app to the Meowgic App Store

## Required information flow

When creating or updating documentation, follow this narrative order.

### 1. What

Explain what the feature, app type, integration, model, or concept is.

Keep this section to one or two short paragraphs.

### 2. Why

Explain why the feature matters.

Focus on:

- The problem it solves
- The result it helps the merchant achieve
- When users should use it
- When users should use a different app type or workflow

### 3. Who

Explain who the feature is designed for.

Use concrete merchant scenarios, store examples, app examples, or comparison tables.

Examples may include:

- A Shopify merchant adding product reviews to a product page
- An operations team creating an internal order management tool
- A personalized-product seller collecting customer images and design data
- A marketing team creating a discount popup without installing another subscription app

### 4. How

Explain how the feature works or how to use it.

Include relevant information about:

- App type
- Supported platform
- Model level
- Credits
- Theme files
- Cloud requirements
- External accounts or credentials
- Configuration
- Previewing and testing
- Publishing
- Store integration
- Updating an existing integration
- Limitations
- Data removal or irreversible actions

For conceptual or reference pages, the How section may explain the workflow at a high level instead of providing step-by-step instructions.

### 5. FAQ

Add frequently asked questions to improve clarity, documentation search, AI retrieval, and organic search visibility.

## Default section order

Use the following heading order unless a section does not apply. Adjust the exact titles to match the topic.

1. Introduction
2. Why and when to use it
3. Common use cases
4. Choose the right app type
5. Prerequisites
6. Create or configure the app
7. Preview and test
8. Publish or integrate
9. Update and manage
10. Limitations
11. FAQ
12. Troubleshooting

Do not add empty sections only to follow the template.

## Content checklist

Before completing a documentation page, verify whether it should include the following information.

### App format

State whether the instructions apply to:

- Section
- Block
- Standalone app
- Multiple app formats

Do not use “app” alone when the app format materially changes the workflow.

### Supported platforms

Specify whether the page applies to:

- Shopify
- SHOPLINE
- Shoplazza
- All supported platforms
- Apps that do not require a connected e-commerce platform

Do not imply that instructions are identical across platforms unless they have been verified.

### Theme requirements

Explain whether the user needs to:

- Upload a theme file
- Select an existing theme
- Provide the active theme
- Work from a duplicated theme
- Add a generated Section or Block through the theme editor
- Upload an updated theme back to the store

Warn users when replacing or uploading a theme could affect existing customizations.

### Cloud requirements

Explain whether Meowgic Cloud is:

- Required
- Recommended
- Optional
- Not needed

Describe why Cloud is needed when applicable, such as:

- Database storage
- Authentication
- File storage
- Real-time data
- Background logic
- AI model access
- App administration
- Shared data across storefront sessions

Do not describe Cloud as required for static storefront features that do not use server-side capabilities.

### Model level

When model selection affects the result, explain the available model levels using outcome-based language.

- **Standard:** Suitable for common storefront features and straightforward app requirements.
- **Advanced:** Suitable for more complex logic, interactions, and higher-quality implementation.
- **Expert:** Suitable for demanding work such as advanced animation, 3D experiences, complex customization, difficult debugging, and full-app inspection.

Do not promise that a higher model level guarantees a specific result.

### Credits and usage

Explain:

- Which action consumes credits
- Whether usage occurs during creation, modification, debugging, or generation
- Whether published-app usage has separate costs
- Whether an external provider charges the user directly
- What happens when the user does not have enough credits or balance

Do not include exact credit amounts unless they are confirmed and current.

### Publishing and versions

Clearly distinguish between:

- The current code version
- The published version
- A version waiting to be pushed
- A version integrated into the store
- A version submitted to the App Store

When relevant, explain whether users must push the latest version before continuing.

### Verification

Include a success indicator whenever possible.

Examples:

- The feature appears in the Shopify theme editor.
- The Block can be added to the product information section.
- The standalone app opens from the published app list.
- Data submitted from the storefront appears in the app dashboard.
- The store is using the latest published version.

## Choosing the right app type

When a page discusses app creation or integration, help users choose the correct format.

| App type | Use it when |
| :-- | :-- |
| Section | The feature should appear as a complete, movable area in the theme editor. |
| Block | The feature should appear inside an existing section, such as product information. |
| Standalone app | The feature is a dashboard, internal tool, management system, generator, landing page, or independently accessible experience. |

Use realistic examples.

### Section examples

- Announcement bar
- FAQ section
- Logo showcase
- Image carousel
- Shoppable video section
- Countdown banner
- Product recommendations

### Block examples

- Product-page upload field
- Size guide button
- Review summary
- Trust badge
- Custom product option
- Delivery estimate
- Product warning
- Add-to-cart enhancement

### Standalone app examples

- Order management dashboard
- Customer support system
- Product data manager
- Image editing tool
- Booking management system
- Order tracking page
- Internal operations tool

## FAQ guidelines

Include three to six FAQs on most documentation pages, depending on the page length and complexity.

Write questions in the natural language users are likely to search or ask Meowgic AI.

Prioritize questions about:

- Supported e-commerce platforms
- Section, Block, and standalone app differences
- Theme compatibility
- Cloud requirements
- Model selection
- Credit usage
- Publishing and integration
- App updates
- Data ownership
- External accounts or API keys
- Limits and quotas
- Removing an integration
- Permanent data deletion
- Common failure cases

Keep answers concise, factual, and easy to scan.

Avoid promotional language in FAQ answers.

Do not repeat information unless the repetition directly answers the question.

## Troubleshooting guidelines

Add a troubleshooting section when users are likely to encounter repeatable problems.

For each issue, explain:

1. What the user sees
2. The most likely cause
3. How to verify the cause
4. How to resolve it
5. What to provide to support if it remains unresolved

Use descriptive issue titles.

Correct:

- The Section does not appear in the theme editor
- The latest app version is not visible in the store
- Cloud cannot be enabled
- Uploaded images do not appear in the dashboard

Incorrect:

- It does not work
- Error
- App problem

Do not suggest destructive troubleshooting steps before safer alternatives.

## Writing principles

### Keep introductions short

The introduction should quickly explain what the feature is and what users can achieve with it.

Do not begin with broad explanations of e-commerce, artificial intelligence, or web development unless they are necessary to understand the feature.

### Focus on outcomes

Explain the merchant outcome before listing technical capabilities.

Prefer:

> Use a product options Block to collect personalization details directly from the product page.

Avoid:

> This feature contains several configurable input components.

### Use concrete examples

Use realistic e-commerce examples instead of generic software examples.

Prefer examples involving:

- Products
- Collections
- Cart behavior
- Storefront design
- Customer data
- Orders
- Product customization
- Marketing campaigns
- Merchant operations

### Keep instructions unambiguous

Use exact UI paths whenever relevant.

Example:

**App → Integration → Shopify → Select theme**

Do not write vague instructions such as:

> Open the relevant settings.

### Explain the result of each important action

Do not only tell users what to click. Explain what changes after the action.

Example:

> Click **Push update**. The latest published version becomes available to the store integration.

### Separate creation from integration

Creating an app does not always mean it has been added to the user’s store.

Documentation must clearly explain when the user is:

- Creating the app
- Publishing the app
- Integrating it
- Activating it in the theme editor
- Updating an existing integration

## SEO and accessibility

- Use descriptive headings that match the terms merchants search for.
- Ensure the frontmatter `title` and `description` clearly explain the page’s purpose.
- Add a `keywords` array when it improves discoverability.
- Use relevant, specific keywords without keyword stuffing.
- Prefer action-oriented page titles.
- Use a shorter `sidebarTitle` when the full page title is too long for navigation.
- Use specific link text instead of “click here.”
- Maintain a logical heading hierarchy.
- Do not skip heading levels.
- Add descriptive alt text to every screenshot, image, video thumbnail, and diagram.
- Describe the purpose of the image, not only what it visually contains.
- Use short paragraphs, lists, tables, and clear section breaks.
- Include the platform name in the title when the workflow is platform-specific.

```text
Integrate a Meowgic app with Shopify

```


## Mintlify components

Use Mintlify components intentionally. Do not use components only for decoration.

### Callouts

Use `<Note>` for additional context that supports the main instructions.

```mdx
<Note>
Publishing an app does not automatically add it to your store theme.
</Note>
```

Use `<Tip>` for recommendations and best practices.

```mdx
<Tip>
Test the integration on a duplicated theme before applying it to your live store.
</Tip>
```

Use `<Warning>` for destructive actions, irreversible steps, security risks, data loss, and critical limitations.

```mdx
<Warning>
Deleting an app may permanently remove its Cloud data and uploaded files.
</Warning>
```

Use `<Check>` to describe a successful result or verification step.

```mdx
<Check>
The new Section now appears in the Shopify theme editor.
</Check>
```

### Steps

Use `<Steps>` and `<Step>` when actions must be completed in order.

```mdx
<Steps>
  <Step title="Select an app">
    Open the app you want to integrate.
  </Step>

  <Step title="Choose Shopify">
    Open **Integration** and select **Shopify**.
  </Step>

  <Step title="Provide your theme">
    Select or upload the theme Meowgic should update.
  </Step>

  <Step title="Test the integration">
    Preview the updated theme and verify that the feature works correctly.
  </Step>
</Steps>
```

Include a verification step whenever possible.

### Tabs

Use `<Tabs>` for platform-specific instructions or mutually exclusive workflows.

Good uses:

- Shopify, SHOPLINE, and Shoplazza
- Section, Block, and standalone app
- Automatic and manual integration
- New integration and update existing integration

Do not use tabs for sequential steps.

### Accordions

Use `<AccordionGroup>` for:

- FAQ
- Troubleshooting
- Edge cases
- Advanced configuration
- Platform-specific limitations

Keep the primary workflow visible. Do not hide required steps inside an accordion.

### Code groups

Use `<CodeGroup>` only when the same example needs to be shown in multiple formats or languages.

Use one code block when only one format is relevant.

### Cards

Use cards when users need to choose between separate destinations or workflows.

Suitable uses include:

- Select an e-commerce platform
- Choose an app type
- Open Cloud, Integration, or Settings documentation

Do not use cards for ordinary sequential instructions.

## Audience and tone

Write primarily for e-commerce merchants, operators, designers, and product teams.

Assume that many readers are not developers.

Explain technical concepts when they affect the user’s decision or workflow, but do not introduce unnecessary implementation details.

Use:

- Active voice
- Second person
- Direct instructions
- Practical examples
- Clear outcomes
- Neutral, confident language

Prioritize clarity, correctness, and usefulness over marketing language.

Avoid describing Meowgic as magical, effortless, unlimited, or guaranteed in instructional content.

Marketing language may appear on landing pages, but not in setup instructions, limitations, billing explanations, or troubleshooting content.

## Style and formatting

### Do not use em dashes

Use a period, comma, colon, or parentheses instead.

Correct:

> Publish the app, then open the integration page.

Incorrect:

> Publish the app—then open the integration page.

### Use active voice

Correct:

> Upload your theme file.

Incorrect:

> The theme file should be uploaded.

Use passive voice only when the state or outcome is more important than the actor.

Example:

> The latest version is now published.

### Use present tense

Use present tense for product behavior and instructions.

Correct:

> Click **Save** to update the app settings.

Use future tense only for a future consequence.

Correct:

> Your unpublished changes will be lost.

### Use sentence case

Use sentence case for page titles and headings.

Correct:

> Integrate with Shopify

Incorrect:

> Integrate With Shopify

Capitalize proper nouns, product names, and acronyms normally.

### Use consistent product terminology

Use:

- Meowgic
- Meowgic AI
- Meowgic Cloud
- Meowgic App Store
- Section
- Block
- Standalone app
- Model level
- Standard
- Advanced
- Expert
- Shopify
- SHOPLINE
- Shoplazza
- Theme
- App
- Task
- Credits

Do not alternate between “plugin,” “extension,” “widget,” and “app” unless those terms have a specific technical meaning in the documented platform.

Do not use “application” and “app” interchangeably on the same page. Prefer “app.”

### UI references

Bold buttons, navigation items, tabs, menu items, and visible UI labels.

Example:

> Open **Settings** and select **Cloud**.

Use inline code for:

- Field names
- Identifiers
- File names
- Configuration values
- URL parameters
- Code values

Example:

> Copy the `app_id` value.

Use `→` for UI paths.

Example:

> **App → Cloud → AI**

Do not place an entire sentence in bold.

### Dates

Use the format:

```text
July 31, 2026
```

Do not use ordinal numbers such as `1st`, `2nd`, or `3rd`.

### Numbers

Use numerals for:

- Credits
- Versions
- File sizes
- Time periods
- Model limits
- Pricing
- Steps when referenced by number

Do not include approximate numbers as confirmed limits.

### Lists

Use bullets for unordered information.

Use numbered lists only when order matters.

Do not create a one-item list.

### Tables

Use tables for comparisons and structured reference information.

Suitable uses include:

- App type comparisons
- Model level comparisons
- Platform differences
- Plan availability
- Status meanings
- Error and resolution mappings

Do not place long procedures inside tables.

## Platform integration pages

Use “platform integration” as the parent term for connecting a Meowgic app to an e-commerce platform.

Use “theme integration” when the workflow specifically modifies or adds functionality to a storefront theme.

Use the platform name when the workflow only applies to one platform.

Examples:

- Shopify integration
- SHOPLINE integration
- Shoplazza integration
- Shopify theme integration

Do not use “store integration” when a more precise term is available.

### Default platform integration page structure

1. Introductory paragraph
2. Supported app types
3. What you need
4. How the integration works
5. Integration steps
6. Preview and testing
7. Activate the feature
8. Update an existing integration
9. Remove the integration
10. Limitations
11. Troubleshooting
12. FAQ

### Introductory paragraph

Explain:

- Which platform the page covers
- Which Meowgic app types can be integrated
- What the merchant can achieve

Do not begin with a general description of the e-commerce platform unless that context is necessary.

### Requirements

State any required:

- Store account
- Theme file
- Theme access
- Staff permission
- App version
- Published version
- Cloud configuration
- Product data
- API credentials

Do not list credentials that the workflow does not require.

### Integration steps

Use exact UI labels and clearly state:

- Which app is being integrated
- Which platform is selected
- Which theme is modified
- Where the feature appears
- How the user verifies it
- Whether additional activation is required in the platform theme editor

### Updates

Explain whether updating the Meowgic app also updates the integrated store automatically.

If the user must push, publish, download, upload, or reintegrate a new version, state this explicitly.

### Removal

Explain the difference between:

- Removing the feature from the theme
- Disconnecting the integration
- Deleting the Meowgic app
- Deleting Cloud data

Do not present these actions as equivalent.

## App Store pages

App Store documentation and app descriptions should help merchants quickly understand what an app does and whether it matches their needs.

Use this order:

1. One-sentence summary
2. Merchant outcome
3. Main capabilities
4. Where the app appears
5. Required configuration
6. Cloud or data requirements
7. Supported platforms
8. Limitations

### One-sentence summary

Describe the app using a concrete e-commerce outcome.

Prefer:

> Let shoppers upload photos and create personalized wall art directly from the product page.

Avoid:

> A powerful and innovative photo customization solution.

### Capability lists

Start each item with an action.

Correct:

- Upload customer images
- Adjust tile size and frame style
- Save customization data
- Add the design to the cart

Avoid inconsistent noun phrases such as:

- Image upload
- Frames
- Data
- Cart

### Do not invent capabilities

Only include capabilities confirmed in the app itself, its screenshots, its configuration, or information provided by the app creator.

Do not infer that an app supports:

- Multiple languages
- Mobile devices
- All themes
- All e-commerce platforms
- Cloud synchronization
- Analytics
- Customer accounts
- Product variants
- Automatic installation

unless that support is confirmed.

## Prompt examples

Place prompt examples in plain-text code blocks.

Use `text wrap` when supported to preserve readability.

A useful prompt should include enough context for Meowgic AI to understand:

- What the user wants to create
- Where the feature should appear
- Who uses it
- What actions users can perform
- What merchants need to manage
- Which platform is involved
- Whether Cloud or stored data is needed
- Any design or behavior requirements

### Section example

```text
Create a product reviews Section for Shopify product pages.

Customers should be able to leave a star rating, write a review, and upload images or videos. Add a dashboard where merchants can approve, edit, hide, and delete reviews.
```

### Block example

```text
Create a delivery estimate Block for the Shopify product information section.

Show the estimated delivery date based on the customer’s location, the current time, and the product’s processing time. Let the merchant configure processing days and excluded dates.
```

### Standalone app example

```text
Create a standalone order tracking app.

Customers should be able to search using a tracking number or an order number and email address. Show the current shipping status, tracking timeline, carrier, and estimated arrival date. Include a dashboard for managing tracking settings.
```

### Theme modification example

```text
Add the generated countdown feature to this Shopify theme as a Section.

Place it on product pages below the product price. Let merchants configure the text, end time, colors, spacing, and expired state from the theme editor.
```

### Prompt-writing rules

- Use direct instructions.
- Describe the desired result instead of prescribing unnecessary technical implementation.
- Include required business logic.
- Include the intended app type when known.
- Include the platform when the task is platform-specific.
- Include the expected management experience when merchants need to configure or manage data.
- Do not add API keys, credentials, or private customer data to public examples.
- Do not promise that copying an example produces an identical result across every theme or platform.
